## 1. 项目概述

### 1.1 背景与目标

**背景**：
- TikTok 矩阵运营需要突破地理限制 + 多账号同时管理
- 现有商业方案（AdsPower、Multilogin）月费高昂，长期运营成本不可控
- 团队具备 AI + 技术能力，适合自研

**目标**：
- 构建支持 200+ 账号同时在线的矩阵运营系统
- 突破 TikTok 地理限制（住宅 IP + 指纹伪装）
- 长期稳定运行，不依赖商业软件月费

**对标参考**：本产品参考 CloakBrowser（架构+反检测）+ Kameleo（部署模式），主要差异为：
- 增量：AI 业务自动化层
- 减量：不从零改 C++ 源码，依赖成熟二进制
- 修改：Go 实例管理层替代商业软件的闭源方案

### 1.2 目标用户

| 用户类型 | 描述 | 核心需求 |
|---------|------|---------|
| 运营团队 | 200 账号矩阵管理 | 多实例同时在线、独立指纹、独立 IP |
| AI 开发团队 | 业务流程自动化 | Python API、MCP 服务、任务调度 |
| 风控团队 | 账号安全管理 | 异常检测、封号预警 |

### 1.3 核心价值主张

**一句话**：用 AI 加速的指纹浏览器矩阵管理工具，实现 TikTok 账号运营的自动化与成本可控。

### 2.2 用户故事

#### US-001: 账号批量注册

**格式**：作为运营团队，我想要批量注册 TikTok 账号，以便快速扩展矩阵规模。

**详细描述**：
- **角色**：运营团队
- **功能**：200 个 TikTok 账号批量注册
- **价值**：减少人工操作，快速起量

**对标参考**：Kameleo 的批量注册流程

**验收标准**：
- AC-001-01：系统能够在 1 小时内完成 50 个账号的注册流程
- AC-001-02：每个账号使用独立指纹和独立 IP
- AC-001-03：注册过程中的验证码自动接收并填写
- AC-001-04：注册失败的账号自动记录并可重试

**边界条件**：
- 输入：Google 账号池、接码平台 API、住宅代理 IP 池
- 输出：已登录的 TikTok 账号凭证
- 异常：接码失败时自动切换号码、IP 被封时自动切换代理

#### US-002: 多实例并行管理

**格式**：作为运营团队，我想要同时管理 200+ 浏览器实例，以便高效运营多个账号。

**详细描述**：
- **角色**：运营团队
- **功能**：200+ 浏览器实例同时运行
- **价值**：一个界面管理全部矩阵

**对标参考**：CloakBrowser 的多实例架构

**验收标准**：

- AC-002-04：实例异常退出不影响其他实例

### 2.4 业务流程

#### Flow-001: 账号注册流程

```
[启动实例] → [分配住宅IP] → [生成指纹] → [打开TikTok注册页]
    ↓
[接收验证码] → [填写验证码] → [完成注册] → [换绑固定号]
    ↓
[保存账号凭证] → [释放临时资源]
```

| 状态 | 描述 | 可转换到 |
|-----|------|---------|
| pending | 等待启动 | starting |
| starting | 正在启动 | running / error |
| running | 运行中 | registering |
| registering | 正在注册 | registered / error |
| registered | 注册完成 | binding_phone |
| binding_phone | 换绑手机号 | completed / error |
| completed | 全部完成 | - |
| error | 发生错误 | pending (重试) |

#### Flow-002: 内容发布流程

```
[AI 请求发布] → [MCP 服务端接收] → [验证账号状态]
    ↓
[选择可用账号] → [上传视频/图文] → [添加描述/标签]
    ↓
[发布成功] → [记录日志] / [发布失败] → [重试或告警]
```

### 2.5 业务规则

#### Rule-001: IP 绑定规则

| 字段 | 内容 |
|-----|------|
| **规则编号** | Rule-001 |
| **规则名称** | 一账号一 IP 绑定规则 |
| **适用实体** | BrowserInstance, ProxyPool |
| **触发条件** | 当创建新的浏览器实例时 |
| **规则内容** | 系统为每个实例分配独立住宅 IP，该 IP 在实例运行期间保持不变，实例销毁后 IP 释放回 IP 池 |
| **优先级** | P0 |
| **关联用户故事** | US-001, US-002 |

#### Rule-002: 指纹一致性规则

| 字段 | 内容 |
|-----|------|
| **规则编号** | Rule-002 |
| **规则名称** | 指纹参数一致性规则 |
| **适用实体** | BrowserInstance |
| **触发条件** | 当生成新的指纹种子时 |
| **规则内容** | 指纹参数（Canvas、WebGL、User-Agent、Timezone、Locale）必须相互一致。例如：Windows UA 不得包含 Apple M1 GPU 信息 |
| **优先级** | P0 |
| **关联用户故事** | US-002 |

#### Rule-003: 验证码重试规则

| 字段 | 内容 |
|-----|------|
| **规则编号** | Rule-003 |
| **规则名称** | 验证码接收重试规则 |
| **适用实体** | 注册流程 |
| **触发条件** | 当接码平台返回失败或超时 |
| **规则内容** | 系统自动重试，最多 3 次，每次间隔 10 秒。3 次失败后切换到下一个可用号码 |
| **优先级** | P1 |
| **关联用户故事** | US-001 |

## 4. 研究分析附录

### 4.1 竞品功能模块对比

#### 模块 1: 指纹引擎

| 竞品 | 指纹定制深度 | 实现方式 | AI 友好度 |
|-----|-------------|---------|----------|
| CloakBrowser | 49 个 C++ 源码补丁 | Chromium 源码修改 | ⭐⭐⭐ |
| Kameleo | 引擎级补丁 | 自研 Chroma/Junglefox | ⭐⭐⭐ |
| AdsPower | 配置级指纹 | 上层配置 | ⭐⭐⭐⭐⭐ |
| **我们的方案** | 源码级 + 上层扩展 | CloakBrowser 二进制 + AI 扩展 | ⭐⭐⭐⭐ |

#### 模块 2: 实例管理

| 竞品 | 多实例支持 | 进程隔离 | API 方式 |
|-----|----------|---------|---------|
| CloakBrowser | ✅ | ✅ | Python/JS |
| Kameleo | ✅ | ✅ | REST API + 多语言 SDK |
| **我们的方案** | ✅ | ✅ | Go 管理层 + MCP |

### 4.2 竞品对比矩阵

| 维度 | CloakBrowser | Kameleo | 我们 |
|-----|-------------|---------|-----|
| **核心功能** |
| 指纹伪装 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 多实例管理 | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| AI 自动化 | ❌ | ❌ | ✅ |
| MCP 支持 | ❌ | ❌ | ✅ |
| **商业模式** |
| 定价 | 免费（开源） | 付费 | 一次性开发 |
| 月成本 | $0 | ~$50/账号 | IP 成本のみ |

### 4.3 最佳实践建议

#### 推荐采用
1. **CloakBrowser 二进制**：开源、成熟、Chromium 生态，内置 49 个反检测补丁，AI 友好
2. **Go 实例管理层**：内存安全、并发好、AI 生成质量高
3. **MCP 协议**：标准 AI 调用协议，生态成熟

#### 建议避免
1. **裸 Chrome/Chromium**：无反检测，容易被 TikTok 风控识别
2. **从零编译 Chromium**：编译 1-2 小时/次，AI 优势被稀释
3. **纯 Python 多进程**：GIL 限制，并发性能差

#### 创新机会
1. **AI + CloakBrowser 集成**：竞品都没有，这是核心差异化
2. **动态指纹策略**：根据 TikTok 风控动态调整 CloakBrowser 指纹参数
